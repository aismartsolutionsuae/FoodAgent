# Runbook — Git/PR workflow (простым языком)

> Зачем: убрать ручную возню и шумные алерты. Решение: DECISIONS.md 2026-05-16
> «Branch protection decomposed». Правило-исполнение: CLAUDE.md → Git / PR workflow.

## Что изменилось в одном предложении

`main` на GitHub не защищён (намеренно). Документы я кладу прямо в `main`.
Код — через Pull Request, который я открываю и веду сам через `gh`. Твоих
кликов в потоке нет.

## 3 сценария — что происходит и когда

### Сценарий 1. Я правлю документ (`*.md`, `docs/**`, `CLAUDE.md`)
- Я коммичу **прямо в `main`**. Никакого PR.
- Алертов нет: `pr-check` для docs не запускается (`paths-ignore`).
- Vercel молчит: проект food-agent удалён, деплоить нечего.
- **Твоё участие: ноль.**

### Сценарий 2. Я правлю код/конфиг
- Я делаю ветку и открываю PR сам: `gh pr create`.
- В **той же сессии** жду проверки: `gh pr checks --watch` (CI ~2–4 мин).
- Зелёно → мержу сам. Красно → не мержу, разбираю, говорю тебе что не так.
- **Твоё участие: ноль.** Никаких кнопок в браузере.

### Сценарий 3. Проверка падает, а сессия уже кончилась
- Я **не** заканчиваю сессию с «висящим» PR.
- Либо домержил при зелёном CI, либо явно сказал:
  «PR #N открыт, CI не дождались — продолжим в новой сессии».
- Telegram-уведомлений пока нет (намеренно — добавим, когда будет реальная
  асинхронная нагрузка, не раньше).

## Автономный агент (`claude-code-action`, ~10% случаев)
Тот, что срабатывает на `@claude` в GitHub без меня в сессии. Он **только
через PR**, в `main` напрямую не пишет. Это единственное место, где риск
«непроверенный код в main» реален — он закрыт на уровне самого агента.

## Что считается «готово» (критерий проверки во фронтенде)
Конструкция считается рабочей только после живой проверки в реальном UI:
1. Документная правка ушла в `main` без PR и без красных алертов.
2. Тестовый PR по коду: я открыл через `gh`, проверки прошли, Vercel-красного
   нет, я смержил — всё без твоих кликов.
3. Ты понимаешь по этим 3 сценариям, что делать НЕ нужно ничего.

Пока пункты 1–3 не подтверждены на практике — считаем конструкцию
непроверенной (см. память: PR-automation pending frontend validation).
